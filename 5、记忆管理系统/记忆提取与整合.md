# Claude Code 记忆提取与融合机制详解

## 概述

Claude Code的记忆管理不是通过简单算法，而是通过**两层模型驱动的流程**：
1. **后台自动提取** - 在每个对话循环结束时，使用forked Claude模型从当前会话的上下文中提取新记忆
2. **定期自动融合** - 使用/dream技能定期整理、去重、融合多个会话积累的记忆

---

## 第一部分：记忆提取流程

### 1. 何时触发提取

**触发点**：`handleStopHooks()` 在query loop完成时调用（stopHooks.ts:149）


- feature gate开启：`feature('EXTRACT_MEMORIES')` = `tengu_passport_quail`
- 自动记忆启用：`isAutoMemoryEnabled()`

**触发频率**：
- 默认：每个query完成时执行

### 3. 相互排斥机制 - 核心创新

**问题**：主代理和后台提取代理都有写入记忆的能力，如何避免冲突？

**解决方案**：`hasMemoryWritesSince()` 检查 (extractMemories.ts:121-148)

```typescript
/**
 * Returns true if any assistant message after the cursor UUID contains a
 * Write/Edit tool_use block targeting an auto-memory path.
 */
function hasMemoryWritesSince(
  messages: Message[],
  sinceUuid: string | undefined
): boolean {
  let foundStart = sinceUuid === undefined
  for (const message of messages) {
    if (!foundStart) {
      if (message.uuid === sinceUuid) foundStart = true
      continue
    }
    if (message.type !== 'assistant') continue

    const content = (message as AssistantMessage).message.content
    if (!Array.isArray(content)) continue

    // 检查是否有Write/Edit工具调用指向auto-memory路径
    for (const block of content) {
      const filePath = getWrittenFilePath(block)
      if (filePath !== undefined && isAutoMemPath(filePath)) {
        return true  // 主代理已写入！
      }
    }
  }
  return false
}
```

**互斥逻辑**：
```typescript
if (hasMemoryWritesSince(messages, lastMemoryMessageUuid)) {
  // 主代理已在此范围写入 → 跳过后台提取
  logForDebugging(
    '[extractMemories] skipping — conversation already wrote to memory files'
  )
  // 向前移动游标，下次提取只看新消息
  const lastMessage = messages.at(-1)
  if (lastMessage?.uuid) {
    lastMemoryMessageUuid = lastMessage.uuid
  }
  return  // 不运行forked agent
}
```

**结果**：
- ✅ 用户显式保存 → 跳过后台提取
- ✅ 用户未保存 → 后台代理填补空缺
- ✅ 避免重复写入和锁冲突

### 5. 消息计数与范围

```typescript
function countModelVisibleMessagesSince(
  messages: Message[],
  sinceUuid: string | undefined
): number {
  if (sinceUuid === null || sinceUuid === undefined) {
    return count(messages, isModelVisibleMessage)  // 全部
  }

  let foundStart = false
  let n = 0
  for (const message of messages) {
    if (!foundStart) {
      if (message.uuid === sinceUuid) foundStart = true
      continue
    }
    if (isModelVisibleMessage(message)) n++  // 仅user + assistant
  }

  // Fallback: 游标丢失时（context compaction后）返回全部
  if (!foundStart) {
    return count(messages, isModelVisibleMessage)
  }
  return n
}
```

**模型可见消息**：仅`user` + `assistant`类型
- ❌ 排除：progress、system、attachment消息
- ✅ 包含：用户输入和助手响应

- 分析显示：~40-80% cache命中率，大大节省token和延迟

#### 6.2 受限权限模型

```typescript
export function createAutoMemCanUseTool(memoryDir: string): CanUseToolFn {
  return async (tool: Tool, input: Record<string, unknown>) => {
    // ✅ 允许
    if (tool.name === REPL_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }

    // ✅ 允许
    if ([FILE_READ_TOOL_NAME, GREP_TOOL_NAME, GLOB_TOOL_NAME].includes(tool.name)) {
      return { behavior: 'allow', updatedInput: input }
    }

    // ✅ 仅read-only Bash（ls, find, grep, cat, stat, wc等）
    if (tool.name === BASH_TOOL_NAME) {
      const parsed = tool.inputSchema.safeParse(input)
      if (parsed.success && tool.isReadOnly(parsed.data)) {
        return { behavior: 'allow', updatedInput: input }
      }
      // ❌ 拒绝write-capable命令
      return denyAutoMemTool(tool, 'Only read-only shell commands allowed')
    }

    // ✅ Edit/Write仅限于memory directory内
    if ((tool.name === FILE_EDIT_TOOL_NAME || tool.name === FILE_WRITE_TOOL_NAME) &&
        'file_path' in input) {
      const filePath = input.file_path
      if (typeof filePath === 'string' && isAutoMemPath(filePath)) {
        return { behavior: 'allow', updatedInput: input }
      }
    }

    // ❌ 拒绝所有其他工具
    return denyAutoMemTool(tool, 'Tool not permitted in extraction context')
  }
}
```

**限制的目的**：
- 不能调用Agent、MCP、web工具
- 不能修改代码或执行任意commands
- 仅能读取和修改自己的memory directory

### 8. 执行和错误处理

```typescript
async function runExtraction({
  context,
  appendSystemMessage,
  isTrailingRun,
}: {
  context: REPLHookContext
  appendSystemMessage?: AppendSystemMessageFn
  isTrailingRun?: boolean
}): Promise<void> {
  inProgress = true
  const startTime = Date.now()
  try {
    // ... 执行forked agent ...
    const result = await runForkedAgent({...})

    // 仅在成功后向前移动游标
    const lastMessage = messages.at(-1)
    if (lastMessage?.uuid) {
      lastMemoryMessageUuid = lastMessage.uuid
    }

    // 提取写入的文件路径
    const writtenPaths = extractWrittenPaths(result.messages)

    // 记录分析指标
    logEvent('tengu_extract_memories_extraction', {
      input_tokens: result.totalUsage.input_tokens,
      output_tokens: result.totalUsage.output_tokens,
      cache_read_input_tokens: result.totalUsage.cache_read_input_tokens,
      message_count: newMessageCount,
      turn_count: turnCount,
      files_written: writtenPaths.length,
      memories_saved: memoryPaths.length,
      duration_ms: Date.now() - startTime,
    })

    // 如果有新记忆，向UI追加消息
    if (memoryPaths.length > 0) {
      const msg = createMemorySavedMessage(memoryPaths)
      appendSystemMessage?.(msg)
    }
  } catch (error) {
    // 提取是best-effort，记录但不通知用户
    logForDebugging(`[extractMemories] error: ${error}`)
    logEvent('tengu_extract_memories_error', {
      duration_ms: Date.now() - startTime,
    })
    // 游标不动，下次重试
  } finally {
    inProgress = false

    // Trailing run处理：如果提取期间又有新context进来...
    const trailing = pendingContext
    pendingContext = undefined
    if (trailing) {
      await runExtraction({
        context: trailing.context,
        appendSystemMessage: trailing.appendSystemMessage,
        isTrailingRun: true,  // 跳过throttle检查
      })
    }
  }
}
```

### 9. 并发管理 - Coalescing

**问题**：多个query快速连续完成，如何处理？

**解决方案**：Stash and trailing run (extractMemories.ts:557-564)

```typescript
if (inProgress) {
  // 已有提取在进行，保存最新context等待尾部run
  logForDebugging('[extractMemories] extraction in progress — stashing')
  logEvent('tengu_extract_memories_coalesced', {})
  pendingContext = { context, appendSystemMessage }  // 覆盖旧context
  return  // 快速返回
}
```

**流程**：
```
Query 1完成 → 启动提取A
Query 2完成（A进行中）→ Stash context2，快速返回
Query 3完成（A进行中）→ 覆盖context2为context3，快速返回
提取A完成 → 触发trailing run，处理context3
Trailing run完成 → 返回
```

**优点**：
- 不阻塞主loop
- 只执行最新的context（不浪费工作）
- inFlightExtractions集合追踪所有promises

## 第二部分：记忆融合流程（/dream）

### 1. 融合的目的

**问题**：随着时间推移，记忆会积累：
- 重复的条目
- 过时的信息
- 互相矛盾的记忆
- 索引(MEMORY.md)变得巨大

**解决方案**：定期运行`/dream`技能，使用模型重新整理所有记忆

### 2. 何时触发融合

#### 2.1 autoDream - 自动触发

触发条件（都要满足，三层门）：

```typescript
// Gate 1: Time-based
const hoursSince = (Date.now() - lastAt) / 3_600_000
if (hoursSince < cfg.minHours) return  // 默认24小时

// Gate 2: Session-based
const sessionIds = await listSessionsTouchedSince(lastAt)
sessionIds = sessionIds.filter(id => id !== currentSession)
if (sessionIds.length < cfg.minSessions) return  // 默认5个会话

// Gate 3: Lock-based
const priorMtime = await tryAcquireConsolidationLock()
if (priorMtime === null) return  // 已有其他进程在consolidating
```

**配置**：
```typescript
const DEFAULTS: AutoDreamConfig = {
  minHours: 24,      // 最少间隔24小时
  minSessions: 5,    // 最少5个会话产生
}
```

#### 2.2 用户显式调用

```bash
/dream
# 或通过/remember技能提示升级记忆到CLAUDE.md
```

### 3. 融合流程的四个阶段

#### 阶段1：定向 (Orient)

```
- ls 记忆目录，看看存在什么
- 读 MEMORY.md 理解当前索引
- 浏览现有主题文件（避免创建重复）
- 如果有 logs/ 子目录（助手模式），查看最近条目
```

**目的**：建立上下文，理解当前状态

#### 阶段2：收集最近信号 (Gather recent signal)

优先级顺序：

```
1. 日志文件 (logs/YYYY/MM/YYYY-MM-DD.md)
   - 如果存在，这是append-only流

2. 现有记忆的漂移
   - 事实与当前codebase矛盾的旧记忆

3. Transcript搜索
   - 仅在怀疑需要时才grep
   - 使用narrow terms搜索
   - 例: grep -rn "build failure" transcripts/ --include="*.jsonl"
```

**指导**：不要详尽读取transcript，仅查找怀疑的内容

#### 阶段3：融合 (Consolidate)

```typescript
// 对每个值得记住的东西，写或更新顶层记忆文件
// 遵循系统提示中的记忆格式和类型约定

// 重点：
// - 将新信号合并到现有主题文件，而不是创建接近重复的
// - 将相对日期("昨天", "上周")转为绝对日期
// - 删除被矛盾的事实 - 如果今天的调查否定了旧记忆，在源处修复它
```

**示例融合操作**：

原有：
```markdown
---
name: Testing strategy
description: Unit tests are fast feedback
type: feedback
---
Focus on unit tests for fast feedback.
```

新信息：来自最近对话 - "集成测试必须使用真实数据库，mock在某次迁移中失败了"

融合后：
```markdown
---
name: Testing strategy
description: Unit tests fast, integration tests with real DB
type: feedback
---
## Unit tests
Fast feedback for business logic.

## Integration tests
**Rule**: Must hit real database, not mocks.
**Why**: Prior incident where mock/prod divergence masked broken migration (2024-Q1)
**How to apply**: All data-layer tests must use real DB or in-memory equivalent
```

#### 阶段4：修剪和索引 (Prune and index)

```
更新 MEMORY.md 保持在 200 行以下 AND ~25KB

- 移除指向现在过时、错误或过时的记忆的指针
- 降级冗长条目：如果索引行超过~200字符，内容应在主题文件中
- 添加指向新重要记忆的指针
- 解决矛盾 — 如果两个文件不同意，修复错误的那个
```

### 5. 去重和更新策略

Claude的记忆融合模型有显式指导：

```
- Merging new signal into existing topic files
  rather than creating near-duplicates
- Converting relative dates ("yesterday", "last week")
  to absolute dates so they remain interpretable
- Deleting contradicted facts —
  if today's investigation disproves an old memory, fix it at the source
```

**关键点**：不是删除旧记忆，而是**合并和更新**

## 第三部分：提取和融合的互作用

### 提取 ↔ 融合的循环

```
Session N:
├─ Query 1...10 → 后台提取 → 新记忆文件A.md, B.md
├─ Query 11...20 → 后台提取 → 更新A.md, 新记忆C.md
└─ Session结束

Session N+1...N+4:
├─ Query ... → 后台提取 → D.md, E.md, 更新B.md
├─ Query ... → 后台提取 → F.md
├─ Query ... → 后台提取 → ...
└─ 没有/dream调用

触发条件满足（24h + 5个session）→ /dream自动运行
┌─ 定向：读A,B,C,D,E,F
├─ 搜索：grep logs/找最近signal
├─ 融合：检测相似的A和E，合并 → AE_merged.md
│        B已有update，不触碰
│        D和F关于同一主题，合并 → DF_merged.md
├─ 修剪：MEMORY.md从35行减至18行
└─ 完成，游标向前

Session N+5:
├─ 新的提取会看到融合后的clean状态
└─ 循环继续...
```

### 关键设计原则

**1. 模型驱动，不是算法**
- 提取：Claude理解上下文，决定什么值得保存
- 融合：Claude识别重复、矛盾、过时，决定如何整理
- 不依赖keyword matching或距离指标

**2. 显式优于隐式**
- 提取前显示现有记忆清单（避免盲目创建）
- 融合时给出四个阶段的显式指导
- 用户可以阅读融合提示，理解逻辑

**3. 非破坏性操作**
- 提取失败？游标不动，下次重试
- 融合进行中？Lock防止并发，失败时回滚
- 用户显式写入？Forked agent检测并跳过

**4. Cache效率**
- Forked agents共享父亲的prompt cache（40-80%命中）
- 减少API调用、token使用、延迟

## 总结表格

| 方面 | 提取 (Extraction) | 融合 (Consolidation) |
|------|------------------|-------------------|
| **何时** | 每个query结束 | 每24h + 5个session |
| **谁执行** | forked Claude | forked Claude |

## 关键创新点

### 1. **Forked Agent 模式**
不是简单的算法，而是使用Claude的推理能力来提取和融合。每个fork：
- 共享prompt cache（大幅节省成本）
- 有受限权限（仅memory dir）
- 有hard turn limit（防止循环）

### 2. **相互排斥设计**
```
主代理写入 ⊕ 后台提取
（同一轮只能一个执行）
```
通过游标和hasMemoryWritesSince()实现，简单优雅。

### 3. **增量游标机制**
每次提取只处理新消息，失败时不丢失进度。
```
提取A: 处理消息1-20，游标→uuid(msg20)
提取B: 处理消息21-40，游标→uuid(msg40)
提取C失败: 游标不动，消息41-60下次重试
提取D: 重新处理消息41-60
```

### 4. **Lock防止融合冲突**
并发融合通过文件锁和mtime追踪防止。
失败时可回滚到上次成功的checkpoint。

### 5. **显式优于隐式**
- 提取前显示现有记忆清单
- 融合时给出四阶段指导
- 用户看得懂prompt，能理解逻辑
