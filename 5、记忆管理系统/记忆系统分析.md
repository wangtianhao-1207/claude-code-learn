# Claude Code 记忆系统设计与实现分析

## 项目概述

Claude Code 2.1.88 实现了一个复杂的、多层次的持久化记忆系统，用于在不同对话会话中跨越时间保留和回忆用户信息、项目上下文和协作指导。该系统是多个专门化模块的集成，共同实现了智能记忆的存储、检索、维护和提取。

---

## 2. 记忆类型系统（Memory Type Taxonomy）

### 2.1 四种记忆类型分类

Claude Code 使用闭合的四类记忆分类法，特意排除了从代码状态可推导的内容：

#### **1. User 记忆**
- **范围**: 总是私有
- **内容**: 用户的角色、目标、责任、知识水平
- **使用场景**:
  - 用户是初学者 vs. 资深工程师 → 调整解释深度
  - 用户的语言偏好和交互风格
  - 用户的技能矩阵（精通Go但新手React）
- **示例**:
  ```markdown
  ---
  name: User expertise profile
  description: User has 10 years Go experience, new to React frontend
  type: user
  ---
  Frame React explanations in terms of backend concepts.
  ```

#### **2. Feedback 记忆**
- **范围**: 默认私有，整个项目的约定时使用团队
- **内容**: 用户关于如何完成工作的指导（避免什么和保留什么）
- **使用场景**:
  - "不要mock数据库，去年迁移测试失败了"
  - "停止总结你做了什么，我能看diff"
  - "单个捆绑PR比多个小PR更好"
- **示例**:
  ```markdown
  ---
  name: Testing strategy
  description: Integration tests must use real database
  type: feedback
  ---
  **Rule**: No mocking databases in integration tests
  **Why**: Prior incident where mock/prod divergence masked broken migration
  **How to apply**: When writing tests for data layer changes
  ```

#### **3. Project 记忆**
- **范围**: 私有或团队，强烈倾向团队共享
- **内容**: 项目内工作、目标、事件、deadline
- **使用场景**:
  - "周四开始冻结合并，移动团队切release分支"
  - "重写auth middleware是因为法规合规要求"
  - "项目截止日期是3月15日"
- **示例**:
  ```markdown
  ---
  name: Release freeze schedule
  description: Mobile team release cut on 2026-03-05
  type: project
  ---
  **Fact**: Merge freeze begins 2026-03-05
  **Why**: Mobile team cutting release branch
  **How to apply**: Flag any non-critical PR work after this date
  ```

#### **4. Reference 记忆**
- **范围**: 通常团队
- **内容**: 外部系统中信息的指针
- **使用场景**:
  - "Linear INGEST项目追踪所有管道bugs"
  - "Grafana仪表板在grafana.internal/d/api-latency"
- **示例**:
  ```markdown
  ---
  name: Bug tracking system
  description: Pipeline bugs tracked in Linear INGEST project
  type: reference
  ---
  Check Linear project INGEST for context on pipeline-related tickets.
  ```

### 2.2 什么不应该保存为记忆

- **代码模式、约定、架构** - 从当前代码状态可推导
- **Git历史** - `git log/blame` 是权威来源
- **调试解决方案** - 修复在代码里，上下文在提交消息
- **CLAUDE.md已记录的内容** - 避免重复
- **临时任务状态** - 使用任务系统而不是记忆

## 3. 记忆提示系统（memdir.ts）

### 3.1 核心函数

#### **buildMemoryLines() - 构建记忆指导**
```typescript
export function buildMemoryLines(
  displayName: string,
  memoryDir: string,
  extraGuidelines?: string[],
  skipIndex = false
): string[]
```

**功能**:
- 生成完整的记忆行为指导（不含MEMORY.md内容）
- 四类记忆类型的详细说明
- 何时保存、如何使用、何时访问的指导
- 返回字符串数组，由调用者组装

**关键指导**:
1. **如何保存**: 两步流程（写到单独文件 + 更新MEMORY.md索引）
2. **索引格式**: `- [标题](文件) — 单行描述`
3. **限制**:
   - MEMORY.md: 最多200行、25KB
   - 单个条目: ~150字符以内
4. **组织原则**: 按主题语义组织，不按时间顺序

#### **buildMemoryPrompt() - 构建完整提示**
```typescript
export function buildMemoryPrompt(params: {
  displayName: string
  memoryDir: string
  extraGuidelines?: string[]
}): string
```

**功能**:
- 调用 `buildMemoryLines()` 获取指导
- 同步读取现有的MEMORY.md内容
- 使用 `truncateEntrypointContent()` 在必要时截断
- 返回完整的记忆提示字符串（包含实际内容）

#### **truncateEntrypointContent() - 智能截断**
```typescript
export function truncateEntrypointContent(raw: string): EntrypointTruncation {
  // 执行行截断和字节截断
  // 记录为什么被截断（用于分析）
}
```

**截断策略**:
1. **行截断**: 先截至200行
2. **字节截断**: 然后截至最后一个换行符前的25KB
3. **警告**: 在截断后追加警告消息
4. **分析**: 记录截断原因和统计数据

### 3.2 KAIROS 模式（助手日志）

对于长期运行的助手会话（feature('KAIROS')）：

```typescript
function buildAssistantDailyLogPrompt(skipIndex = false): string
```

**差异**:
- **写入**: 追加到日期命名的日志文件而不是维护MEMORY.md
- **路径**: `~/.claude/memory/logs/YYYY/MM/YYYY-MM-DD.md`
- **格式**: 时间戳化的短条目（append-only）
- **维护**: 单独的 `/dream` 技能在夜间将日志提炼成主题文件 + MEMORY.md

### 3.3 目录初始化

```typescript
export async function ensureMemoryDirExists(memoryDir: string): Promise<void>
```

**作用**:
- 幂等地创建记忆目录
- 由 `loadMemoryPrompt()` 调用（每个会话一次）
- 使模型可以直接写文件（无需检查存在性）
- 失败时继续，FileWriteTool会报真正的权限错误

## 4. 记忆扫描与检索（memoryScan.ts + findRelevantMemories.ts）

### 4.1 内存文件扫描

```typescript
export async function scanMemoryFiles(
  memoryDir: string,
  signal: AbortSignal
): Promise<MemoryHeader[]>
```

**扫描流程**:
1. **递归读取**: 扫描memoryDir下所有.md文件（排除MEMORY.md）
2. **并行读取**: 使用Promise.allSettled读取frontmatter头
3. **排序**: 按修改时间从新到旧排序
4. **限制**: 最多200个文件（MAX_MEMORY_FILES）
5. **返回**: MemoryHeader数组，包含文件名、路径、mtime、描述、类型

**MemoryHeader结构**:
```typescript
type MemoryHeader = {
  filename: string           // 相对路径
  filePath: string          // 绝对路径
  mtimeMs: number          // 修改时间戳
  description: string | null // frontmatter中的描述
  type: MemoryType | undefined // 记忆类型
}
```

### 4.2 记忆清单格式化

```typescript
export function formatMemoryManifest(memories: MemoryHeader[]): string
```

**输出格式**:
```
- [type] filename (ISO-timestamp): description
- [feedback] testing_strategy.md (2026-03-17T14:30:00.000Z): Integration tests must use real DB
- [project] sprint_schedule.md (2026-03-16T09:15:00.000Z): Sprint ends 2026-03-22
```

**用途**:
- 提示中显示给模型（recall选择器）
- 提取记忆时显示（自动提取代理）

### 4.3 相关记忆查找

```typescript
export async function findRelevantMemories(
  query: string,
  memoryDir: string,
  signal: AbortSignal,
  recentTools: readonly string[] = [],
  alreadySurfaced: ReadonlySet<string> = new Set()
): Promise<RelevantMemory[]>
```

**流程**:
1. **扫描**: 调用 `scanMemoryFiles()` 获取所有记忆
2. **过滤**: 排除已显示过的（alreadySurfaced）
3. **选择**: 调用 `selectRelevantMemories()` 使用Sonnet选择
4. **返回**: 最多5个最相关的记忆（带mtime）

#### **选择算法**:

使用Claude Sonnet作为侧查询（side query）：

```
SELECT_MEMORIES_SYSTEM_PROMPT:
"你正在为Claude Code选择有用的记忆。根据用户查询和可用记忆列表返回最多5个记忆的文件名。只包含肯定有帮助的。不确定时不包含。"
```

**参数**:
- model: Sonnet（轻量级）
- max_tokens: 256
- output_format: JSON with selected_memories array
- querySource: 'memdir_relevance'（用于分析）

**优化**:
- 排除最近使用的工具的参考文档
- **但保留**: 关于这些工具的警告、陷阱、已知问题

## 9. /remember 技能（remember.ts）

### 9.1 目的

```typescript
description: 'Review auto-memory entries and propose promotions to
CLAUDE.md, CLAUDE.local.md, or shared memory. Also detects outdated,
conflicting, and duplicate entries across memory layers.'
```

### 9.2 四个步骤

#### **Step 1: 收集所有记忆层**
- 读取项目根目录的 CLAUDE.md 和 CLAUDE.local.md
- 检查auto-memory（已在系统提示中）
- 识别团队记忆部分（如果有）

#### **Step 2: 分类每个auto-memory条目**

| 目标 | 属于谁 | 示例 |
|------|-------|------|
| CLAUDE.md | 所有贡献者都应遵循的Claude指导 | "使用bun不npm", "API路由使用kebab-case" |
| CLAUDE.local.md | 特定用户的个人指导 | "我偏好简洁响应", "不要auto-commit" |
| Team memory | 跨repo的组织范围知识 | "PR通过#deploy-queue", "staging at staging.internal" |
| Stay in auto-memory | 不确定或临时的工作笔记 | 会话特定的观察 |

#### **Step 3: 识别清理机会**
- **重复**: auto-memory中已在CLAUDE.md/local.md中的内容
- **过期**: 被较新auto-memory条目矛盾的CLAUDE文件条目
- **冲突**: 任意两层之间的矛盾

#### **Step 4: 呈现报告**
分组按操作类型：
1. **晋升** - 要移动的条目，带目标和理由
2. **清理** - 重复、过期、冲突
3. **不确定** - 需要用户输入的条目
4. **无需操作** - 应留在原处的条目

### 9.3 关键规则

```typescript
const SKILL_PROMPT = `
- 呈现所有提议后再做任何更改
- 不要在没有明确用户批准的情况下修改文件
- 关于不确定的条目询问，不要猜测
`
```

## 11. 团队记忆（TEAMMEM 功能）

### 11.1 两层架构

### 11.2 类型变体

- 无<scope>标签

## 14. 记忆与其他持久化机制

### 14.1 区别

| 机制 | 作用域 | 何时使用 |
|------|-------|---------|
| **Memory** | 跨会话持久 | 未来对话中有用的信息 |
| **Plan** | 单一对话 | 非平凡实现前对齐方法 |
| **Tasks** | 单一对话 | 追踪当前工作的步骤 |
| **Session Memory** | 单一会话 | 长会话中的上下文摘要 |
| **CLAUDE.md** | 项目范围 | 所有贡献者应遵循的约定 |

### 14.2 记忆与Plan

```
When to use Plan instead of memory:
- 即将开始非平凡实现任务且想与用户对齐方法时
- 已有计划但改变方法时，通过更新计划而不是保存记忆来持久化

When to use memory:
- 信息对未来对话有用
- 用户显式要求记住
```

### 14.3 记忆与Tasks

```
When to use Tasks instead of memory:
- 需要将当前对话工作分解为离散步骤时
- 需要在当前对话内追踪进度时

When to use memory:
- 信息对未来对话有用
- 工作需要在会话边界上保留
```

## 总结

Claude Code的记忆系统是一个精心设计的多层次架构：

| 层级 | 组件 | 作用 |
|------|------|------|
| **1. 存储** | FileSystem + frontmatter | 持久化+元数据 |
| **2. 类型** | 4-类分类法 | 语义组织 |

| **8. 管理** | /remember + /memory | 用户控制 |

这个系统在**无需用户干预的自动化**和**显式用户控制**之间取得平衡，同时通过保守的截断、新鲜度警告和验证指导来维护安全性和准确性。
